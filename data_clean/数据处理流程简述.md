# data_clean 数据处理流程简述

## 1. 当前入口

现在 `data_clean` 已经分成两条入口：

- 微信公众号线
- 教务数据库线

两条线共用后半段的抽取、判重、核验、回流和入库逻辑，但前半段输入不同。

## 2. 微信公众号线

输入位置：

- 开发态：`data/data_clean/input/wechat/`
- Jarvis：`/root/data_process/data/data_clean/input/wechat/`

输入对象：

- `RawPage`

核心字段：

- `page_id`
- `source_type`
- `source_name`
- `source_url`
- `title`
- `raw_html_path`
- `published_at`
- `fetched_at`

处理流程：

1. 读取 `RawPage`
2. 读取 HTML
3. 规则清洗为 `original_text`
4. 进入 `LLM1 -> 召回 -> LLM2 -> 规则校验 -> LLM3 -> 回流/入库`

## 3. 教务数据库线

输入源：

- `/root/data_process/data/seu_campus_assistant.db`
- 表：`school_event_table`

实际使用字段：

- `id`
- `website`
- `title`
- `cleaned_document`

输入对象：

- `JwcDbRecord`

映射规则：

- `record_id = id`
- `page_id = jwc_db_<id>`
- `website = website`
- `title = title`
- `original_text = cleaned_document`

这条线不再依赖：

- `raw_html_path`
- `published_at`
- `fetched_at`
- `html_cleaner.py`

处理流程：

1. 从源库读取教务记录
2. 直接把 `cleaned_document` 作为 `original_text`
3. 进入 `LLM1 -> 召回 -> LLM2 -> 规则校验 -> LLM3 -> 回流/入库`

## 4. 输出位置

最终事件目标库：

- Jarvis：`/root/rivermind-data/database/jiaowu.db`

私有运行时：

- 开发态：`data_clean/runtime/`
- Jarvis：`/root/data_process/data_clean/runtime/`

其中包括：

- `process_state.sqlite3`
- `raw_cache/`
- `logs/`

## 5. 当前状态说明

现在不是“还没接真实数据结构”，而是：

- 微信公众号线仍然是文件输入型
- 教务线已经切成数据库输入型
- 最终写回库已从原来的 `data_clean/output/seu_campus_assistant.db` 切换为 Jarvis 上的 `jiaowu.db`

一句话概括：

- **微信线走 `RawPage + HTML`**
- **教务线走 `DB record + cleaned_document`**
- **两条线共用后半段清洗入库链路**
