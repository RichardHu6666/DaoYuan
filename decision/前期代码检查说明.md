# Decision 前期代码检查说明

本文档面向当前 `decision` 决策层的前期开发阶段。

适用场景：
- 决策层代码已经写好，但真实数据库、真实事件数据、API key 还没有完全到位
- 需要先做“代码级自检”和“运行环境预检”
- 需要在云端 `Jarvis` 的 `/root/decision` 下保留一份可直接执行的说明

## 1. 当前目录约定

云端部署路径默认约定如下：

- 决策层代码：`/root/decision`
- 数据库 helper：`/root/rivermind-data/seu_campus_db_v2.py`
- 决策层数据库：`/root/rivermind-data/database/seu_campus_assistant.db`

本地开发仓库中的对应 helper 文件是：

- `E:\RichardHu6666\CampusJarvis\data\seu_campus_db_v2.py`

## 2. 当前已经提供的检查入口

当前 `decision` 模块支持两个调试入口：

### 2.1 直接响应调试

```bash
cd /root
python3 -m decision respond <user_id> "<instruct>"
```

兼容旧写法：

```bash
cd /root
python3 -m decision <user_id> "<instruct>"
```

说明：
- 这个入口会实际进入 `ClientManager -> ClientSession -> DecisionEngine`
- 依赖真实运行环境
- 如果数据库、环境变量或 embedding 依赖未准备好，可能直接失败

### 2.2 前期代码检查

```bash
cd /root
python3 -m decision check
```

说明：
- 这是“无真实数据也能跑”的预检入口
- 默认只做代码级检查，不要求真实数据库文件存在
- 内部会使用 fake gateway / fake llm / fake retriever 跑一轮内存 smoke

默认检查内容包括：
- `decision` 包能否正常导入
- 默认配置约束是否自洽
- `tool_direct` 主链路是否正常
- `rag` 主链路是否正常
- `ClientManager.release_idle_clients()` 是否会 flush context

如果输出类似下面这样，就说明前期代码层面通过：

```text
overall=PASS
[PASS] code_imports: ...
[PASS] code_config_contract: ...
[PASS] code_in_memory_smoke: ...
```

## 3. 运行环境预检

如果除了代码级检查，还想顺手检查运行时路径、数据库 helper、环境变量准备度，可以执行：

```bash
cd /root
python3 -m decision check --runtime
```

如果希望结构化输出，方便后续脚本或 agent 读取：

```bash
cd /root
python3 -m decision check --runtime --json
```

这个模式会额外检查：
- 配置加载结果是否合理
- `db_helper_path` 指向的文件是否存在
- helper 中是否真的有 `SEUCampusDB`
- 数据库父目录是否存在
- 数据库文件是否存在
- LLM API key 是否已配置
- embedding 依赖是否准备好

## 4. 如何理解检查结果

### 4.1 `check` 通过

这表示：
- 决策层代码结构没有明显断裂
- session / engine / retriever / tool route 的主干逻辑可以在 fake 依赖下跑通
- 当前适合继续做代码重构、接口补全、提示词调整、单元测试扩展

这不表示：
- 真实数据库已经可用
- 真正的 embedding 调用已经可用
- 真正的线上响应已经可用

### 4.2 `check --runtime` 出现 `warn`

这通常表示“现在还没准备好真实运行环境”，常见且允许，例如：
- 数据库文件还没有初始化
- API key 还没有注入
- 本地没有装 `sentence_transformers`

前期阶段这些 `warn` 是可以接受的。

### 4.3 `check --runtime` 出现 `fail`

这通常说明存在真正会阻断后续接真环境的问题，例如：
- `db_helper_path` 指向错误
- helper 文件里没有 `SEUCampusDB`
- `final_k > recall_k`
- 配置数值非法

这种 `fail` 需要先修掉，再进入真实环境联调。

## 5. 当前推荐的检查顺序

在还没有真实数据时，建议按这个顺序做：

1. 跑代码级预检

```bash
cd /root
python3 -m decision check
```

2. 如果后续要接真环境，再跑运行时预检

```bash
cd /root
python3 -m decision check --runtime --json
```

3. 等真实数据库和环境变量准备好之后，再跑真实调试请求

```bash
cd /root
python3 -m decision respond test_user "最近有什么适合我的竞赛？"
```

## 6. 后续接入真实数据前，需要确认的最小条件

至少确认下面几项：

- `/root/rivermind-data/seu_campus_db_v2.py` 存在
- `/root/rivermind-data/database/` 目录存在
- `/root/rivermind-data/database/seu_campus_assistant.db` 已初始化
- `DEEPSEEK_API_KEY` 或 `DECISION_LLM_API_KEY` 已注入
- 如果使用 `openai` embedding：
  - `OPENAI_API_KEY` 或 `DECISION_EMBEDDING_API_KEY` 已注入
- 如果使用 `local_bge_m3`：
  - 本机可导入 `sentence_transformers`

## 7. 当前代码层已覆盖的核心验证点

当前单测和预检已覆盖这些重点：

- 同用户请求串行化
- 空闲 session 回收与 context flush
- 工具直达分流
- LLM1 异常时降级到 RAG
- 缺失画像字段时的推荐分支
- 检索时跳过非法向量

因此，在真实数据尚未到位的阶段，`decision check` 可以作为第一道“是否还能继续开发”的守门命令。

## 9. 一句话结论

在真实数据还没来的这段时间，优先使用：

```bash
python3 -m decision check
```

它的目标不是验证“线上是否可用”，而是先验证“决策层代码主干是不是完整、稳定、可继续开发”。
