# Decision 调用说明

本文档只回答一个问题：

`decision` 这一层现在到底应该怎么调用。

适用对象：
- 你自己手动调试
- 后续其他 agent 接入
- 未来把它接到 `FastAPI`、`NoneBot`、Web 服务或其他上层入口

## 1. 当前已经存在的调用入口

当前 `decision` 层已经有 3 种直接可用的调用方式：

1. 命令行单次调用
2. Python 直接调用 `ClientManager`
3. Python 直接调用 `DecisionEngine`
4. FastAPI 常驻服务调用

推荐优先级：

1. 正式低延迟接入：用 `serve`
2. 调试单条请求：用命令行
3. 做真实会话管理：用 `ClientManager`
4. 做更底层实验：用 `DecisionEngine`

## 2. 命令行调用

最直接的调试方式：

```bash
cd /root
python3 -m decision respond <user_id> "<instruct>"
```

例子：

```bash
cd /root
python3 -m decision respond test_user "最近有什么适合我的竞赛？"
```

兼容旧写法：

```bash
cd /root
python3 -m decision test_user "最近有什么适合我的竞赛？"
```

这个入口内部实际做的事情是：

1. 创建一个 `ClientManager`
2. 用 `user_id` 取到对应 `ClientSession`
3. 调用 `client.respond(instruct)`
4. 返回最终文本响应
5. 关闭 manager，并在需要时 flush context

适合场景：
- 手动 smoke test
- 看当前环境能不能跑通
- 快速验证一条 query 的结果

不适合场景：
- 高频多轮对话服务
- 长生命周期 session 管理
- Web 接口正式接入

## 3. FastAPI 常驻服务调用

正式低延迟调用建议使用常驻服务，而不是每次都重新执行一次 `python3 -m decision respond ...`。

启动方式：

```bash
cd /root
python3 -m decision serve --host 127.0.0.1 --port 8000
```

说明：

- 服务进程内长期持有一个 `ClientManager`
- 同一个 `user_id` 会复用同一个 `ClientSession`
- 当前部署必须保持单进程、单 worker
- 不要使用多 worker 模式共享 session

接口：

1. 健康检查

```bash
curl http://127.0.0.1:8000/api/health
```

2. 对话调用

```bash
curl -X POST http://127.0.0.1:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"user_id":"test_user_full","instruct":"最近有什么讲座推荐？"}'
```

3. 清空会话

```bash
curl -X POST http://127.0.0.1:8000/api/chat/clear \
  -H "Content-Type: application/json" \
  -d '{"user_id":"test_user_full"}'
```

这里才是正式低延迟入口；`respond` 仍然保留，但只适合单条 smoke test。

## 4. 前期检查调用

如果现在还没有真实数据，先不要急着跑真实请求，先跑检查：

```bash
cd /root
python3 -m decision check
```

如果要连运行环境一起检查：

```bash
cd /root
python3 -m decision check --runtime --json
```

这两个命令不是对话接口，而是“调用前自检接口”。

## 5. Python 推荐调用方式

如果你后续要从服务代码里正式接入，推荐直接调用 `ClientManager`，不要每次都自己手搓 session 生命周期。

这里最关键的一点是：

- `user_id` 是在“创建/获取 client”的时候传入的
- 不是在每次 `client.respond()` 的时候重复再传一遍

也就是说，这一层的语义应该理解成：

```python
client = await manager.get_client(user_id)
reply = await client.respond(instruct)
```

而不是：

```python
reply = await client.respond(user_id, instruct)
```

示例：

```python
import asyncio

from decision import ClientManager


async def main():
    manager = ClientManager()
    try:
        client = await manager.get_client("user_001")
        reply = await client.respond("最近有什么适合我的竞赛？")
        print(reply)
    finally:
        await manager.close()


asyncio.run(main())
```

这一层会自动帮你处理：
- `user_id -> ClientSession` 的映射
- 同用户请求串行化
- context 加载
- context 写回
- 空闲 session 清理

## 6. 更底层的调用方式

如果你不想走 `ClientManager`，也可以直接调用 `DecisionEngine`。

示例：

```python
import asyncio

from decision import DecisionEngine


async def main():
    engine = DecisionEngine()
    context = []
    result = await engine.respond(
        user_id="user_001",
        instruct="最近有什么适合我的竞赛？",
        context=context,
    )
    print(result.response_text)
    print(result.route)
    print(result.retrieved_event_ids)


asyncio.run(main())
```

这里返回的不是纯字符串，而是 `DecisionResult`。

当前常用字段：
- `response_text`
- `route`
- `tool_name`
- `tool_url`
- `rewritten_query`
- `retrieved_event_ids`
- `missing_profile_fields`

适合场景：
- 做底层调试
- 单元测试
- 观察路由结果和召回结果

不适合场景：
- 直接拿来做多用户会话服务

## 7. 调用链路到底长什么样

当前主链路是这样的：

```text
上层调用
-> ClientManager.get_client(user_id)
-> ClientSession.respond(instruct)
-> DecisionEngine.respond(user_id, instruct, context)
-> 工具直达 / LLM1 路由 / RAG 检索 / LLM2 生成
-> 返回 DecisionResult 或 response_text
```

如果你要的是“用户看到的最终文本”，就取：

- `await client.respond(instruct)` 的返回值

如果你要的是“调试信息也一起拿到”，就直接调：

- `await engine.respond(...)`

## 8. 上层接入时建议怎么封装

推荐方式是：

1. 服务进程内持有一个长生命周期 `ClientManager`
2. 每次请求到来时按 `user_id` 获取 `ClientSession`
3. 调用 `await client.respond(instruct)`
4. 服务关闭时统一 `await manager.close()`

也就是说，上层接口最适合封成这种样子：

```python
reply_text = await manager.get_client(user_id).respond(instruct)
```

严格写法需要拆两步，因为 `get_client()` 本身也是 `await`：

```python
client = await manager.get_client(user_id)
reply_text = await client.respond(instruct)
```

## 9. 当前输入输出约定

### 输入

如果你走 `ClientManager / ClientSession` 这一层，输入是分两步的。

第一步，创建或获取 client：

- `user_id`

第二步，让这个 client 响应一条消息：

- `instruct`

也就是说，`user_id` 不属于每次 `respond()` 的入参，它属于 client 绑定阶段。

如果直接走更底层的 `DecisionEngine`，那才是一次性显式传两个参数：

- `user_id`
- `instruct`

其中：
- `user_id` 用来绑定用户画像和 context
- `instruct` 是用户自然语言请求

### 输出

如果走 `ClientSession.respond()`：
- 返回 `str`

如果走 `DecisionEngine.respond()`：
- 返回 `DecisionResult`

## 10. 当前调用前必须知道的事

如果你跑的是：

```bash
python3 -m decision respond ...
```

那它依赖真实运行环境，至少要保证下面这些东西基本到位：

- `DECISION_DB_HELPER_PATH` 对应的 helper 可导入
- `DECISION_DB_PATH` 对应数据库可访问
- LLM key 可用
- embedding 配置可用

如果这些条件还没齐，先跑：

```bash
python3 -m decision check
```

## 10. 现在最推荐你怎么用

### 场景 A：还没真实数据

先用：

```bash
python3 -m decision check
```

### 场景 B：想手动试一条真实 query

用：

```bash
python3 -m decision respond test_user "最近有什么适合我的竞赛？"
```

### 场景 C：后续服务正式接入

用：

```python
manager = ClientManager()
client = await manager.get_client(user_id)
reply = await client.respond(instruct)
```

### 场景 D：你要看内部路由和召回细节

用：

```python
engine = DecisionEngine()
result = await engine.respond(user_id=user_id, instruct=instruct, context=context)
```

## 11. 一句话结论

如果你只关心“现在直接怎么调”，就记住这三条：

1. 代码预检：`python3 -m decision check`
2. 单条请求调试：`python3 -m decision respond <user_id> "<instruct>"`
3. 正式服务接入：`ClientManager -> ClientSession.respond()`
